# Mangox Audit — June 2026

Scope: (1) iOS 27 / Swift surface we aren't exploiting, (2) WHOOP & Strava API state vs. latest releases, (3) performance overall and in the AI chat. Verified against the current `main` tree (iOS 27 SDK, Swift 6.0 mode, strict concurrency = complete, warnings-as-errors).

## TL;DR

The codebase is already advanced. On-device coach uses `LanguageModelSession` streaming with `PartiallyGenerated`, `DynamicProfile`, `reasoningLevel`, the Spotlight tool, third-party `LanguageModel` providers (Anthropic/Google), image attachments, `prewarm`, `tokenCount` budgeting, and `useCase: .contentTagging`. Chat rendering isolates per-token streaming into its own view and memoizes markdown. WHOOP is already on **v2**. Strava is **v3** with token-in-header.

So the real backlog is short and specific:

| # | Item | Severity | Effort |
|---|------|----------|--------|
| P0 | Strava base URL migration `www.strava.com/api/v3` → `www.api-v3.strava.com` (mandatory by **2027-06-01**) | High (hard deadline) | S |
| P0 | Strava 2026 Developer Program tiering — confirm app is registered under the right tier (Standard needs Strava sub, ≤10 athletes; Extended otherwise) | High (access risk) | S (ops) |
| P1 | Adopt FM `response.usage` token/cost tracking (new iOS 27) — surface third-party billing + PCC budget telemetry. Wrapper placed, gated on `sdkExposesResponseUsage` flag (absent beta-1). | Medium | S |
| P1 | WHOOP v2 **webhooks** instead of 4h polling — push recovery/sleep/workout updates | Medium | M |
| P2 | Use WHOOP `read:sleep`/`read:workout` scopes we already request but don't consume | Low | M |
| P2 | FM system `OCRTool` — scan gym-machine displays / paper plans / race results into the coach. Gated on `sdkExposesOCRTool` flag (absent beta-1). | Low | M |
| P2 | Migrate homegrown coach eval harness → real iOS 27 **Evaluations** framework | Low | M |
| P3 | Build-time: split the >2.5k-line view/service files; consider Swift 6.4 language mode | Low | M |

## 1. iOS 27 / Swift — what we already use, and the gaps

### Already adopted (no action)
- `LanguageModelSession.streamResponse` + `@Generable` `PartiallyGenerated` streaming — `OnDeviceCoachEngine.swift:510,602`.
- `DynamicProfile` with `.model()` / `.reasoningLevel()` — `CoachDynamicProfiles.swift`.
- Third-party `LanguageModel` protocol providers (Anthropic Claude / Google Gemini via SPM) — `MangoxCoachLanguageModelProviders.swift`.
- Image attachments (`Transcript.ImageAttachment`) — `CoachImageAttachmentSupport.swift`.
- Spotlight system tool — `MangoxOnDeviceCoachTools.swift`.
- `SystemLanguageModel(useCase: .contentTagging)`, `tokenCount` snapshot budgeting, `prewarm`.
- Swift 6.0 language mode, `SWIFT_STRICT_CONCURRENCY = complete`, `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`, approachable concurrency, member-import-visibility upcoming feature.

### Gaps worth taking
1. **`response.usage` (new in iOS 27).** `respond`/streamed results now expose `usage.input.totalTokenCount`, `usage.input.cachedTokenCount`, `usage.output.totalTokenCount`, `usage.output.reasoningTokenCount`. We currently estimate budget with `tokenCount` up-front but never read actual usage. Wins: real cost surfacing for the paid Claude/Gemini tier, PCC reasoning-token visibility, and tighter snapshot sizing from measured (not estimated) input tokens.
2. **Evaluations framework (new).** `FoundationModelsCoachEvaluation.swift` is an "Evaluations-framework-style" homegrown harness (golden fixtures + validators). The actual iOS 27 Evaluations framework gives statistical regression scoring across prompt tweaks — migrate the routing/narrow-reply fixtures into it to catch coach regressions with confidence intervals.
3. **`OCRTool` system tool.** We use the Spotlight tool but not the Vision-backed `OCRTool`. Natural Mangox fit: photograph a gym ERG/treadmill summary screen, a paper training plan, or a race-result printout → OCR → coach ingests. Pairs with the image-attachment path we already have.
4. **PCC `reasoningLevel(.deep)` selectivity.** We reference `reasoningLevel` in profiles — confirm `.deep` is reserved for plan generation / hard analysis and `.light` for chat, since deep reasoning burns reasoning tokens and latency. (Verify in `CoachDynamicProfiles.swift:129`.)
5. **Swift 6.4 language mode (WWDC26).** Optional. Theme is "less boilerplate." Low payoff but cheap once on the iOS 27 toolchain; gate behind a green build since warnings are errors.

> SwiftUI's iOS 27 perf improvements (faster state init / layout) are free on recompile — no code change required.

## 2. WHOOP & Strava

### WHOOP — already on v2 ✅, but polling-only
- Endpoints in `WhoopService.swift` are all `/developer/v2/...` (`v2/user/profile/basic`, `v2/user/measurement/body`, `v2/cycle`, `v2/cycle/{id}/recovery`, `v2/user/access`). v1 was fully removed (webhooks gone 2026-05-31), so **we're clear of the deprecation.**
- **Gap — no webhooks.** Refresh is poll-based every 4h (`refreshLinkedDataIfStale`). WHOOP v2 supports webhooks for recovery/sleep/workout updates. Moving to webhooks gives near-real-time readiness on the Home screen without battery-cost polling. (Requires a webhook receiver — fits the existing Supabase edge-function used for OAuth token exchange.)
- **Gap — requested-but-unused scopes.** We request `read:sleep read:workout read:cycles` but only consume recovery + body measurement. Either consume sleep/strain (richer coach context) or drop the scopes to minimize the consent surface.

### Strava — v3, two real deadlines
The 2026 Developer Program changes are the important news.

- **P0 — base URL migration (hard deadline 2027-06-01).** Strava is moving the API host to `https://www.api-v3.strava.com`. We hardcode `https://www.strava.com/api/v3/...` in **8 places** in `StravaService.swift` (uploads, athlete, athlete/activities, activities/{id}, photos). Token-in-header is **already** how we auth (`Authorization: Bearer`), so that half of the 2027 change is satisfied — only the base URL remains.
- **P0 (ops) — tiering.** Standard Tier = self-managed, **≤10 athletes**, and **requires an active Strava subscription** for the developer. Extended Access Tier = higher limits / more users, no sub required, but needs approval. Confirm which tier the Mangox client ID is registered under and that it matches the user base, or API access can be cut.
- **Not affected:** Club Activities/Members/Admins and Segments-Explore deprecations (effective 2026-09-01) — we don't call them. The `segments` references in `InstagramStoryCardRenderer.swift` are route/chart segments, not the Strava Segments API.
- Current rate limits unchanged: 200 req / 15 min, 2,000 / day (100 / 1,000 non-upload).

## 3. Performance

### AI chat — already well-architected
- **Per-token isolation:** `CoachStreamingSection` (`CoachConversationView.swift:735`) reads all streaming state in its own `body`, so token updates re-render only the pending bubble, never the full transcript. This is the correct pattern.
- **Markdown memoization:** `CoachAssistantFormatting.cachedAttributedContent` (NSCache, 256 entries) parses markdown + runs the metric-highlight regex once per committed message instead of on every invalidation.
- Committed transcript uses `LazyVStack`; streaming uses `streamResponse` snapshots.

Remaining chat nits (low severity):
- Cache key is the full message body string (`"\(category)\u{1F}\(raw)"`). Fine at current scale; if transcripts get very long, key on a hash to bound key memory.
- `applyMetricHighlighting` maps offsets via `String.distance` per match (O(n) per metric). Only runs on the cached path, so acceptable — leave it.
- Confirm the pending bubble does **not** re-run full markdown parsing every token — streaming should use `attributedContentForStreaming` (lighter, no metric tinting), which it does.

### Overall app
- **Build time / type-check:** several files are very large — `AIService.swift` (3,824), `OutdoorDashboardView.swift` (3,793), `SummaryView.swift` (2,548), `DashboardView.swift` (2,521), `ConnectionView.swift` (2,470), `SettingsDetailViews.swift` (2,468). Giant SwiftUI `body`s are the usual cause of slow incremental type-checking. Splitting the largest views into subview structs improves both compile time and runtime diffing. (No correctness issue — purely velocity/perf.)
- 305 Swift files in a single module; module boundaries are physical only. Already tracked in the 2.0 rebuild plan.
- Free win: recompiling against the iOS 27 SDK picks up SwiftUI's layout/state perf improvements with zero code change.

## 4. Plan — implementation & fix steps

### P0 — Strava (deadline-driven)
1. In `StravaService.swift`, introduce one `private static let apiBase = URL(string: "https://www.api-v3.strava.com")!` (mirroring WHOOP's `apiBase` pattern) and replace all 8 `www.strava.com/api/v3` literals with `apiBase.appendingPathComponent(...)`. Leave `oauth/token` + `oauth/authorize` on `www.strava.com` (OAuth host is unchanged, same as WHOOP).
2. Add an integration test hitting the new host (or a unit test asserting every request URL begins with `api-v3`) to lock the migration in.
3. **Ops:** log into the Strava developer portal, confirm the Mangox app's tier and that the developer account has the required subscription (Standard) or Extended approval. Document the tier in `Docs/`.

### P1 — FM usage telemetry
1. In `OnDeviceCoachEngine`/`AIService`, after each non-streamed `respond` and at the end of each stream, read `response.usage` and record `{input.total, input.cached, output.total, output.reasoning}`.
2. Surface paid-tier (Claude/Gemini) token counts in the provider settings screen for cost transparency; feed measured input tokens back into the snapshot-budget heuristic in `MangoxPCCSupport` to replace the `~1900 / ~8000` estimates with real numbers.

### P1 — WHOOP webhooks
1. Add a webhook receiver to the existing Supabase edge-function stack; register the WHOOP v2 webhook subscription with the same client credentials.
2. On `recovery.updated` / `workout.updated`, push a lightweight signal that triggers `WhoopService.refreshLinkedData()` (or updates cached recovery directly), and relax the 4h poll to a fallback.

### P2 — depth items
- **Sleep/strain:** decode WHOOP `v2/activity/sleep` + cycle strain into `WhoopService` and expose to the coach snapshot; or drop unused scopes.
- **OCRTool:** register `OCRTool` in `MangoxOnDeviceCoachTools`, wire it to the existing image-attachment input so a photographed machine/plan/result becomes structured coach context.
- **Evaluations framework:** port `FoundationModelsCoachEvaluation` fixtures into the iOS 27 Evaluations framework; run in CI to gate coach prompt changes.

### P3 — velocity
- Split the six >2.4k-line files into subview/extension files (start with `OutdoorDashboardView` and `AIService`).
- Trial `SWIFT_VERSION = 6.4` once on the iOS 27 toolchain; keep only if the build stays warning-clean.
